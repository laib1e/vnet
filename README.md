# vnet - заметки по разработке

## Проблема: неправильный порядок байт в IP-адресах.
- После загрузки модуля (sudo insmod vnet.ko) и создания маршрута (sudo ip route add 10.0.0.1/32 dev vnet0)
- При попытке пропинговать интерфейс ожидается правильный порядок байт IP-адреса в логах модуля ядра (sudo dmesg | grep "vnet")
- Попробовал `ip_hdr(skb)` — тот же результат
- Попробовал ручное смещение через `skb->data + sizeof(ethhdr)` — тот же результат
- Гипотеза: возможно Ethernet-заголовка нет и IP начинается с другого смещения
- Следующий шаг: hex-дамп первых 34 байт skb->data

hex-дамп
[  314.164077] vnet: skb len = 42, data first 34 bytes:
[  314.164082] vnet: 00000000: ff ff ff ff ff ff 00 00 00 00 00 00 08 06 00 01  ................
[  314.164085] vnet: 00000010: 08 00 06 04 00 01 00 00 00 00 00 00 c0 a8 d1 82  ................
[  314.164086] vnet: 00000020: 00 00 

Разбор:
- Байты 12-13: 08 06 - это EtherType. 0x0806 - это ARP, а не IP (0x0800)
- MAC адрес отсутствует, первые 6 байт FF, означает широковещательный

Ядро хочет отправить ICMP пакет на 10.0.0.1, но сначала ему нужно узнать MAC-адрес получателя. 
Поэтому оно шлет ARP-запрос. Ответа нет и до ICMP дело не доходит. Код вызывающий ошибку интерпретировал ARP-запрос как IP-заголовок поэтому порядок байт был перепутан.

Было принято решение для виртуального интерфейса отключить ARP.
